<objective>
Harden install.sh and stow.sh into a strictly zero-prompt, hardware-guarded, re-runnable installer pair that produces the same end state on a genuinely fresh Arch system as on the dev machine, and that hard-fails loudly (never silently) when a package does not land.

Purpose: Satisfies INST-01 (correct package names + post-install verification), INST-02 (stow completes on a fresh system, no unguarded operations), and the package-list half of CLEAN-01 (remove the retired launcher and the phantom `scripts` entry). This is the "installer actually works on a fresh box" slice — the reproduction gate in 03-04 exercises exactly these two scripts.
Output: install.sh restructured into flagged sections with hardware guards, orphan/limine/bootstrap guards, and a hard-fail verification table; stow.sh made idempotent with a guarded config move, non-interactive shell switch, and a first-boot theme seed.
</objective>

## Phase Goal (MVP user story)

**As a** developer reproducing this desktop on a fresh Arch system, **I want to** run install.sh + stow.sh unattended and get a fully themed desktop from a clean repo, **so that** the entire rice reproduces from scratch with one workflow and no manual fixups.

This plan delivers the slice that makes the two setup scripts survive a real fresh box: no aborts, no prompts, and a loud failure the instant a package is missing.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI args → install.sh | `--core-only`/flags are untrusted input; an unrecognized flag must not silently alter behavior |
| package layer → running system | pacman/AUR place packages the rest of the setup assumes exist; a silent miss (adw-gtk3 ghost) breaks theming invisibly |
| installer → bootloader/config on disk | limine.conf and hyprland.conf are pre-existing system state a naive mv/rm can destroy or abort on |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-02-DOS | Denial of Service | unguarded mv/rm/orphan-removal aborting a fresh/re-run | high | mitigate | Existence/type guards on hyprland.conf mv, orphan-removal count guard, limine rm -f with prior backup (Tasks 2-3); a partial run no longer half-configures the box |
| T-03-02-VAL | Tampering (input validation) | install.sh flag parser | low | mitigate | Unknown flags print to stderr and exit nonzero; only `--core-only`/`--help` accepted (Task 1, Security V5) |
| T-03-02-BOOT | Tampering (data destruction) | `sudo rm /boot/limine/limine.conf` | high | mitigate | Back up to limine.conf.bak before removal; `rm -f` idempotent; block guarded by a limine-presence check — no path to an unbootable box from a re-run |
| T-03-02-INTEG | Spoofing/Tampering | silent package install failure | medium | mitigate | verify_packages hard-fail table over every package in the selected sections (D-65) turns a silent miss into a blocking nonzero exit |
| T-03-SC | Tampering | AUR/pacman package supply chain | medium | accept | No new npm/pip/cargo packages; swaync moves to official `extra`; all AUR names were re-verified via paru -Si in RESEARCH (Phase-1 T-01-SC precedent); integrity delegated to the pacman keyring GPG chain |
</threat_model>
