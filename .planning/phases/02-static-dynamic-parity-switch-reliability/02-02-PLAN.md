## Phase Goal

**As a** dotfiles maintainer, **I want to** prove that 10 consecutive theme switches with Thunar and Walker open leave every app correctly themed on the final switch, **so that** I can trust repeated real-world switching never drifts, sticks white, or serves a stale cache.

> MVP framing note: verification-and-hardening phase; the "user" is the maintainer. This plan delivers the repeated-switch reliability capability (PIPE-06) end-to-end: a runnable stress harness → automated per-switch verdicts → a human eyes-on final sign-off (the non-negotiable evidence bar for this "stuck-white"-scarred project).

<objective>
Deliver the PIPE-06 reliability proof: a rerunnable, parameterized `theme-stress-test` harness that drives the real `theme-apply` entrypoint through 10 alternating static↔dynamic switches with Thunar and Walker open, asserts per-switch correctness (theme-doctor pass, format-normalized sentinel match, walker+elephant health), aborts on first failure with diagnostics, fixes any reliability bug it surfaces in Phase-1 engine code (D-40), and closes on a clean full gate (D-41) plus a human visual sign-off recorded in 02-VERIFICATION.md.

Purpose: Prove the Phase-1 engine stays correct under repeated real use — the requirement that most directly rebuts this project's long history of looping theme-fix commits.
Output: `theme-stress-test`, any engine-code reliability fixes it forces, a passing stress log + passing parity log, and 02-VERIFICATION.md with the human sign-off and the D-15/D-37 caveat documented.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI arg → filesystem path | A theme-name-bearing argument (via flags) is interpolated toward `palettes/$name.json`. |
| tool stderr → notify-send | Stress diagnostics could reach a user-facing notification. |
| harness → live session processes | The harness kills/relaunches Thunar/walker — a broad kill would DoS the user's own session. |
| stress logs → state dir | New logs/ artifacts under the 700 state dir. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-04 | Tampering | theme-stress-test theme-name arg | high | mitigate | Reuse theme-apply's validate-against-actual-palette-filenames guard before any path interpolation; reject unknown names. |
| T-02-05 | Denial of Service (own session) | harness process management | high | mitigate | Only narrowly-scoped `pgrep -x` / `killall -q <named-binary>` targeting thunar/walker; reuse the bounded-poll + targeted-kill patterns hardened in lib/reload.sh + lib/gtk.sh — no unscoped killall. |
| T-02-06 | Information Disclosure | stress notifications | medium | mitigate | Truncate+strip any stderr reaching notify-send (`head -c 200 \| tr -d`); full detail stays in the log. |
| T-02-07 | Information Disclosure | ~/.local/state/theme/logs/ | low | mitigate | Create logs/ under the existing chmod 700 state dir and inherit; do not loosen perms. |
