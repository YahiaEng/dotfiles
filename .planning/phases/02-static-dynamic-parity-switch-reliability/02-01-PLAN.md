## Phase Goal

**As a** dotfiles maintainer, **I want to** prove — with rerunnable tools — that static presets and matugen dynamic themes render an identical output contract, **so that** I can trust one theme switch re-themes the whole desktop the same way in both modes and catch any palette/template regression before it ships.

> MVP framing note: this is a verification-and-hardening phase; the "user" is the maintainer running the tools. Each task advances a runnable end-to-end proof capability (manifest → tool → verdict), not a horizontal layer. Plan 02-01 delivers the parity capability (PIPE-04); Plan 02-02 delivers the repeated-switch reliability capability (PIPE-06).

<objective>
Deliver the PIPE-04 parity proof: a canonical output-contract manifest (`contract.json`) plus a rerunnable, render-only `theme-parity` checker that proves all 7 render targets (6 static presets + materialyou) share identical file structure, identical per-file variable/key-name sets, and well-formed color values — then make `theme-doctor` read the same manifest so the file list has one source of truth.

Purpose: Phase 1 built the shared engine; this plan proves both modes are genuinely one pipeline with an identical output contract, and hardens any divergence it finds (D-40).
Output: `contract.json`, `lib/contract.sh`, `theme-parity`, a refactored `theme-doctor`, and a clean all-green parity log under `~/.local/state/theme/logs/`.
</objective>
