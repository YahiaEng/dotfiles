<objective>
Close the FIX-02 UAT gap (Test 2 / D-23): pressing ENTER on the empty password field before typing causes a failed-auth loop where subsequent keystrokes never register.

Root cause (fully diagnosed — see debug session): hyprlock 0.9.5 silently DROPS all keyboard input while a PAM verification is in flight. Pressing ENTER on the empty field starts such a verification because `general:ignore_empty_input` defaults to 0 and is unset here — an empty password is submitted to PAM, pam_unix applies its ~2s failure delay, and every character typed during that ~2-3s blocked window vanishes with no visual feedback.

The fix is config-only, in `hypr/.config/hypr/hyprlock.conf`:
1. Add `ignore_empty_input = true` to `general{}` so ENTER on an empty buffer is ignored outright — no PAM round, no blocked window, no faillock tally growth (closes the reported gap).
2. Add an input-field `check_text` cue so any legitimate in-flight verification (e.g. a genuinely wrong password) is a visible "checking" state rather than a silent dead keyboard (defense-in-depth for the remaining wrong-password blocked window).

Both option names are pre-verified against the installed hyprlock 0.9.5 binary schema, so neither will be silently ignored — the exact trap that caused 04-02.

Purpose: Restore 100% first-try unlock on the ENTER-first path, matching the already-working type-immediately path.
Output: Migrated `hyprlock.conf`; a documented human re-UAT protocol (D-23) for end-of-phase verification.
</objective>
