# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Scope Note (read first)

Phase 4 is a **diagnostic/reliability phase**, not a redesign phase. Per `04-CONTEXT.md` (`<domain>` and D-16) and `ROADMAP.md`, this phase fixes three defects (wlogout shutdown hang, hyprlock first-keystroke drop, kitty startup performance) and closes one packaging gap (rsync in `install.sh`). **No new visual design is introduced.** The actual wlogout and hyprlock redesigns are explicitly deferred to Phase 6 (WLOG-01, LOCK-01). This project also has no component library, no `components.json`, and no web/frontend tech stack (verified: no `package.json`/`components.json`/`tailwind.config.*` in repo root) — the shadcn initialization gate does not apply.

This UI-SPEC therefore functions as a **visual-parity and interaction-integrity contract**: it defines what must stay pixel-for-pixel/behaviorally identical while the underlying reliability bug is fixed, plus the few copy/content elements this phase does touch (fastfetch greeting, action labels). Sections that don't apply to a non-redesign phase are marked N/A with rationale rather than populated with invented values, per this agent's scope boundary.

**Not applicable — no new palette work.** Both wlogout and hyprlock already source all colors live from the shared theme pipeline (`~/.local/state/theme/hyprland.conf`, resolved through matugen or the static preset). This phase must not hardcode, override, or introduce any new color value in either surface.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | inherited from `~/.local/state/theme/hyprland.conf` | unchanged — existing theme-pipeline output |
| Secondary (30%) | inherited from `~/.local/state/theme/hyprland.conf` | unchanged — existing theme-pipeline output |
| Accent (10%) | inherited (`$primary`/`$secondary`/`$tertiary` per hyprlock.conf) | unchanged — hyprlock time label (`$primary`), date label (`$on_surface`), greeting (`$secondary`), auth-check (`$tertiary`), auth-fail (`$error`), capslock (`$secondary`) |
| Destructive | `$error` (existing, hyprlock auth-fail only) | unchanged — no new destructive-action color introduced this phase |

Accent reserved for: the existing hyprlock label/auth-indicator set listed above only. No new accent usage is introduced. If `wleave` replaces `wlogout` (FIX-01, D-15), it must consume the same `~/.local/state/theme/` state-dir pattern via its own `style.css`/theming hook so the rendered colors are visually indistinguishable from the current wlogout output — verify by side-by-side screenshot comparison before/after the swap, not by re-deriving a new palette.

## Copywriting Contract

This is the one section with real content this phase touches — action labels (wlogout audit, D-14) and the fastfetch greeting (D-01/D-02, refined by RESEARCH.md).

| Element | Copy |
|---------|------|
| Primary CTA | not applicable — no new user-facing button/action is introduced this phase |
| wlogout action labels (6, unchanged) | Lock · Logout · Suspend · Hibernate · Shutdown · Reboot — keybinds `l`/`e`/`u`/`h`/`s`/`r` stay as-is. If `wleave` replaces wlogout (D-15), migrate this exact label/keybind set — do not rename, reorder, or drop an action. |
| Fastfetch greeting frame | Box-drawing frame (`┌─...─┐` / `├─...┤` / `└─...┘`) and section grouping (user → distro/kernel/pkgs/uptime → wm/term/shell/font → cpu/gpu/mem/disk/display → colors) stay visually identical. Only individual **module removal** is in scope (D-02), and only evidence-gated by `fastfetch --stat` at execution time — not a blind trim. Candidate modules per RESEARCH.md's direct read of this repo's actual `config.jsonc`: `disk` (stats `/`) and `gpu` (shells out on NVIDIA) — **not** DisplayManager/battery/power, which CONTEXT.md named generically but do not exist in this repo's current config. Do not remove `os`/`kernel`/`packages`/`uptime`/`wm`/`terminal`/`shell`/`terminalfont`/`cpu`/`memory`/`display`/`colors` — those are cheap and not implicated. |
| Empty state heading/body | not applicable — no empty-state UI this phase |
| Error state | not applicable — no new user-facing error UI this phase; diagnosis errors surface via `journalctl`/`coredumpctl`/terminal output only, consumed by the executor, not end-user-facing copy |
| Destructive confirmation | not applicable — none of the six wlogout actions gain a new confirmation dialog this phase (existing behavior, e.g. any current confirm-on-shutdown prompt in `style.css`/layout, stays exactly as shipped) |
| hyprlock grace-period behavior change (interaction note, not copy) | `grace = 0` (per D-18) removes the current "move cursor during grace period to unlock without a password" convenience. This is a **behavior** change, not a copy change — no on-screen text changes, but document it in the phase SUMMARY per D-25 so the user isn't surprised the grace-unlock shortcut is gone. |
| Lockout-recovery procedure (D-20) | Internal documentation, not in-app copy — written to phase planning artifacts before first hyprlock test (second-TTY-login + `pkill hyprlock` escape hatch), reused by Phase 6 LOCK-01. Not a UI element. |
